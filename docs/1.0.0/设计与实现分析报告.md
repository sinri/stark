## 2. 项目定位与总体设计

### 2.1 定位

项目定位是一个基于 Vert.x 的轻量 Java 框架，强调：

- 不依赖 Spring 生态
- 以 Verticle 作为主要运行单元
- 提供可替换日志体系（含 SLF4J 桥接、SLS 输出）
- 提供队列/定时任务/HTTP 服务等基础组件

对应文档入口：`README.md`。

### 2.2 模块划分

- `core`
  - `Stark`：对 `Vertx` 的包装（`VertxWrapper`）并扩展异步工具能力
  - `StartAsyncMixin`：异步循环、迭代、步进、sleep 等统一能力
  - `LateObject`：一次性延迟初始化容器
- `component.verticles`
  - `StarkVerticleBase`：统一 Verticle 生命周期封装
- `component.web`
  - `HTTPService`：HTTP server 启停、路由配置钩子
- `component.queue`
  - `QueueDispatcher` + `QueueTask` + `QueueWorkerPoolManager`
- `component.sundial`
  - `CronJobDispatcher` + `CronExpression`
- `database.mysql`
  - 基于 Vert.x MySQL client 的池封装和注册表
- `logging`
  - 自定义日志对象、渲染器、处理器、SLF4J provider、SLS putter

## 3. 设计质量评估

### 3.1 优点

- 架构思路清晰：围绕 Verticle 生命周期组织组件，利于扩展。
- `StarkVerticleBase` 对部署状态、`undeployed()` Future 做了统一抽象，降低业务 Verticle 样板代码。
- `StartAsyncMixin` 提供了可复用的异步循环/批处理语义，API 组织完整。
- 日志链路有明确的“Log -> Render -> Processer”分层，替换点明确。
- JSpecify 注解在多个包中使用，空值语义比普通 Java 项目更清楚。

### 3.2 主要设计风险

- 生命周期控制存在“看似有主流程、实际上未阻塞等待”的问题（`StarkProgram`）。
- 队列调度器停止后仍会继续定时递归调度，存在后台任务泄漏风险。
- 关键外部集成（SLF4J provider、SLS 上传）存在错误语义不一致或初始化时序问题。
- 当前测试覆盖集中在 core/logging；对 queue/sundial/http/mysql/sls 的行为回归覆盖明显不足。

## 4. 实现质量评估（按模块）

### 4.1 Program 启动主流程

`StarkProgram.delegateMain` 采用 Future 串联启动各组件，流程可读性好，但存在主线程等待缺失问题：

- 创建了 `CountDownLatch`，但方法内未 `await`。
- 结果是“逻辑上准备等待服务结束”，实现上却直接返回。

该问题对框架使用者影响大（尤其命令行主程序场景）。

### 4.2 Queue 组件

优点：

- 调度器与任务对象拆分清晰；并发控制由 `QueueWorkerPoolManager` 负责。

风险点：

- `QueueDispatcher.routine()` 在 `eventually` 中无条件 `setTimer` 下一轮，停止后仍会持续自调度。
- `QueueTask.startVerticle()` 内部异步链路 fire-and-forget，`startVerticle` 立即返回成功，生命周期与实际执行脱钩。
- worker 计数只在 `undeployMe().onSuccess` 里减少；失败路径会导致“并发计数卡住”。

### 4.3 Cron 调度组件

优点：

- `CronExpression` 解析支持 `*`、范围、步进、枚举，满足分钟级任务调度基础需求。

风险点：

- `CronJobDispatcher` 首轮执行前不会先 `refreshPlans()`，导致启动后首个调度窗口可能拿不到最新计划。
- 高频 debug 日志（每分钟对每个 plan 都打“匹配/不匹配”）在计划规模增大时会放大日志成本。

### 4.4 Logging 与 SLF4J 桥接

优点：

- 自定义日志结构（`Log`）和渲染器扩展点完整。

关键风险：

- `StarkSlf4jServiceProvider` 使用 `LateObject` 懒初始化，但在真实运行中 `getMDCAdapter()` 可能在 `initialize()` 前被调用，导致 NPE。
- 该问题在本次执行 `mvn test -q` 时已出现启动告警栈追踪。

### 4.5 SLS 日志输出

- `AliyunSLSLogPutter.putLogsImpl` 对 HTTP 非 200 和网络异常统一吞掉，最终返回成功 Future。
- 上游无法基于 Future 判断真实投递结果，可靠性监控困难。

### 4.6 MySQL 池封装

- 采用全局静态注册表便于快速获取连接池。
- 但缺少 unregister/close 协同与生命周期治理，长期运行或多环境切换可能引入资源管理问题。

## 5. 测试与工程化现状

## 6. 结论

Stark 1.0.0 的基础架构方向是正确的，代码可读性和模块边界整体较好，已具备轻量框架雏形；但当前版本存在若干高风险生命周期/可靠性问题，且测试覆盖集中在非核心运行路径。

如果要进入可稳定对外使用状态，建议优先修复：

1. 启动主流程阻塞/退出语义（`StarkProgram`）
2. 队列调度停止后的递归定时泄漏（`QueueDispatcher`）
3. SLF4J provider 初始化时序 NPE（`StarkSlf4jServiceProvider`）
4. SLS 上传错误语义（失败应可感知）

详见同目录问题清单：`docs/1.0.0/问题清单.md`。
