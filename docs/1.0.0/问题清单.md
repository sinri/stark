# Stark 1.0.0 问题清单

> 说明：按严重级别排序（P0/P1/P2/P3）。

## P0

### 1. SLF4J Service Provider 初始化时序可触发 NPE

- 状态：已修复（2026-04-05，已复检）
- 位置：
  - `src/main/java/io/github/sinri/stark/logging/slf4j/StarkSlf4jServiceProvider.java:14-16`
  - `src/main/java/io/github/sinri/stark/logging/slf4j/StarkSlf4jServiceProvider.java:29-30`
- 现象：`getMDCAdapter()` 直接读取 `LateObject`，当 SLF4J 在 provider `initialize()` 之前调用 getter 时，会抛 `NullPointerException`。
- 复现证据：执行 `mvn test -q` 时出现 `SLF4J(E): Failed to instantiate SLF4J LoggerFactory`，堆栈中包含 `LateObject has not been initialized`。
- 影响：SLF4J 绑定不稳定，第三方库（如 Netty/Vert.x）初始化期日志行为不可预期。
- 建议：
  1. 去除 `LateObject`，在字段声明处直接初始化 `BasicMDCAdapter/BasicMarkerFactory/StarkSlf4jLoggerFactory`；或
  2. getter 中做惰性兜底初始化，保证任何调用时刻都返回非空对象。
- 复检结果：执行 `mvn test -q` 后，不再出现 `SLF4J(E)` 与 `LateObject has not been initialized`。

### 1.1 剩余观察（低优先级）

- 位置：`src/main/java/io/github/sinri/stark/logging/slf4j/StarkSlf4jServiceProvider.java:14`、`20`
- 说明：`getLoggerFactory()` 仍依赖 `LateObject`。在标准 SLF4J SPI 路径下通常先 `initialize()` 再调用该 getter，风险较低；当前不作为阻断问题处理。

### 2. QueueDispatcher 在 undeploy 后仍可能继续递归调度

- 状态：已修复（2026-04-05，已复检）
- 位置：
  - `src/main/java/io/github/sinri/stark/component/queue/QueueDispatcher.java:107-113`
  - `src/main/java/io/github/sinri/stark/component/queue/QueueDispatcher.java:185-188`
- 发生条件：
  1. `QueueDispatcher` 已经部署并进入 `routine()` 递归；
  2. 外部执行 `undeploy`（触发 `stopVerticle()`）；
  3. 此前某一轮 `routine()` 已在 `eventually` 中排入下一次 `setTimer`。
- 现象：
  1. `stopVerticle()` 将状态改为终止态后，后续定时回调仍会触发 `routine()`；
  2. 实例理论上已卸载，但仍持续产生日志和信号读取行为；
  3. 在重启/滚动发布时，旧实例可能与新实例并行轮询队列。
- 影响：
  1. 生命周期语义错误（已 stop 的实例仍有活动）；
  2. 可能出现重复调度、资源泄漏、日志噪声；
  3. 在多实例交替期间增加重复消费风险。
- 边界说明：仅“读到业务 `STOP` 信号后继续轮询下一轮”本身属于可接受设计（用于等待后续 `RUN`），不算该缺陷。
- 建议：
  1. 增加“停止标记 + 当前 timerId 记录与取消”；
  2. 在 `eventually` 调度前区分“业务 STOP”与“生命周期 STOP（undeploy）”，后者不再递归；
  3. 为 undeploy 场景补测试（断言 stop 后不再触发下一轮 routine）。
- 修复说明：
  1. 状态机调整为 `INIT/RUNNING/PAUSED/TERMINATED`，其中 `TERMINATED` 仅由 `stopVerticle()` 进入；
  2. 增加 `TERMINATED` 不可逆保护；
  3. 增加下一轮 routine timer 的记录与取消，`TERMINATED` 后不再调度。

## P1

### 3. StarkProgram 主流程未阻塞等待，`CountDownLatch` 无实际作用

- 状态：已修复（2026-04-06，已复检）
- 位置：`src/main/java/io/github/sinri/stark/program/StarkProgram.java:22`、`103-114`、`117`
- 现象：创建并 `countDown` 了 `latch`，但没有 `await`。
- 影响：主流程语义与实现不一致；`delegateMain` 结束时不保证服务生命周期已受控，易造成调用方误用。
- 建议：
  1. 若设计为阻塞式启动，需在方法末尾 `await` 并处理中断；
  2. 若设计为非阻塞式，删除 `latch` 并在 API 文档明确行为。
- 修复说明：
  1. `delegateMain` 已在启动编排末尾调用 `latch.await()`，主流程改为阻塞等待服务结束或启动失败；
  2. 方法签名改为 `throws InterruptedException`，中断语义由调用方显式处理。

### 4. QueueTask 运行生命周期与 Verticle 生命周期脱钩

- 状态：设计确认（2026-04-06，非问题）
- 位置：`src/main/java/io/github/sinri/stark/component/queue/QueueTask.java:58-77`
- 现象：`startVerticle()` 启动异步链后立即返回 `Future.succeededFuture()`，框架认为启动已完成；实际 `run()` 在后台继续执行。
- 结论：该行为已按“异步 fire-and-forget 任务模型”确认为有意设计，不作为缺陷处理。
- 注意事项：调用方与运维侧应避免将 Verticle deploy 成功等同于任务完成，建议在业务层补充任务级状态/结果观测。

### 5. Queue worker 计数在 undeploy 失败时可能泄漏

- 位置：`src/main/java/io/github/sinri/stark/component/queue/QueueTask.java:74`
- 现象：`whenOneWorkerEnds()` 仅在 `undeployMe().onSuccess` 调用；失败时不减计数。
- 影响：长时间运行后可能出现并发计数偏高，导致 `QueueWorkerPoolManager.isBusy()` 长期为真，队列吞吐下降甚至停滞。
- 建议：在 `onComplete` 中统一归还计数，并单独记录 undeploy 失败。

### 6. SLS 上传错误被吞掉，调用方无法感知失败

- 状态：设计确认（2026-04-06，非问题）
- 位置：`src/main/java/io/github/sinri/stark/logging/aliyun/sls/putter/AliyunSLSLogPutter.java:139-152`
- 现象：HTTP 非 200 仅打印日志后返回成功；网络异常 `recover` 后也返回成功。
- 结论：该行为已确认为有意设计（日志链路以“尽量不影响主业务流程”为优先），不作为缺陷处理。
- 注意事项：如果未来需要“强一致日志投递”，建议在上层增加独立可靠投递通道或告警补偿机制。

## P2

### 7. CronJobDispatcher 启动后首轮计划刷新滞后

- 状态：已修复（2026-04-06，已复检）
- 位置：`src/main/java/io/github/sinri/stark/component/sundial/CronJobDispatcher.java:48-53`
- 现象：仅在 periodic 回调内执行 `refreshPlans()`，启动到首轮回调期间 planMap 可能为空。
- 影响：服务启动后的首个调度窗口可能错过应执行任务。
- 建议：`startVerticle()` 里先执行一次 `refreshPlans()`，再进入周期调度。
- 修复说明：
  1. `startVerticle()` 已改为先执行 `refreshPlans()`；
  2. 首次刷新成功后才注册 `setPeriodic`，降低启动初期漏调度风险。

### 8. SlsLogProcesser 队列无上限，存在内存增长风险

- 位置：`src/main/java/io/github/sinri/stark/logging/aliyun/sls/putter/SlsLogProcesser.java:21`、`39-43`
- 现象：`ConcurrentLinkedQueue` 无容量限制，`process()` 无背压/丢弃/降级策略。
- 影响：下游写入变慢时，内存可持续增长。
- 建议：增加可配置上限与策略（阻塞、丢弃、采样、落盘）。

### 9. MySQL 池静态注册表缺少生命周期治理

- 位置：`src/main/java/io/github/sinri/stark/database/mysql/StarkMySQLPool.java:11-22`
- 现象：全局静态 map 支持注册/读取，但无 unregister 与关闭协同。
- 影响：多次重建连接池或测试环境切换时，可能残留旧对象引用。
- 建议：补充 unregister API、幂等 close 约束，以及注册名冲突策略。

## P3

### 10. 命名和文案遗留不一致

- 位置：`src/main/java/io/github/sinri/stark/component/web/HTTPService.java:74`
- 现象：日志文案仍为 `KeelHttpServer Exception`（旧项目名痕迹）。
- 影响：排障与品牌一致性较差。
- 建议：统一替换为 Stark 语义命名。

### 11. 测试覆盖结构失衡

- 证据：`target/surefire-reports` 显示 69 个用例集中在 `core` 和 `logging`。
- 缺口模块：`queue`、`sundial`、`web`、`database/mysql`、`aliyun/sls`。
- 影响：当前高风险路径缺少回归保护。
- 建议：
  1. 先补 P0/P1 涉及模块的最小回归测试；
  2. 增加含失败路径的集成测试（尤其队列停止、SLS 非 200、SLF4J provider 初始化顺序）。
